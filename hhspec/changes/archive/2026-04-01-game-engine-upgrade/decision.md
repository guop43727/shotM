# 关键决策：game-engine-upgrade

## 决策 1：Phaser 3 版本选择
- **选择**：Phaser 3.80.1（最新稳定版）
- **理由**：
  - 最新版本包含 WebGL 性能优化和 bug 修复
  - 社区活跃，文档完善，问题响应快
  - 向后兼容性好，未来升级路径清晰
- **备选**：Phaser 3.55 LTS（长期支持版）
  - 优点：更稳定，适合长期维护项目
  - 缺点：缺少最新性能优化和新特性
  - 否决原因：本项目处于快速迭代期，需要最新特性支持

## 决策 2：构建工具配置
- **选择**：Vite 5.x + ES 模块
- **理由**：
  - 开发服务器启动快（< 1s），热更新响应快（< 100ms）
  - 原生支持 ES 模块，无需复杂配置
  - 生产构建使用 Rollup，产物体积小
  - 与 Phaser 3 生态兼容性好
- **备选**：Webpack 5
  - 优点：生态成熟，插件丰富
  - 缺点：配置复杂，构建速度慢
  - 否决原因：开发体验不如 Vite，配置成本高

## 决策 3：模块化方案
- **选择**：ES 模块 + 命名导出
- **理由**：
  - 符合现代 JavaScript 标准
  - Tree-shaking 友好，减少产物体积
  - 类型推导更准确（配合 JSDoc）
  - 便于后续引入 TypeScript
- **备选**：保持全局变量模式
  - 优点：无需改动现有代码
  - 缺点：命名空间污染，依赖关系不清晰
  - 否决原因：不利于长期维护和扩展

## 决策 4：渐进式迁移阶段划分
- **选择**：5 阶段迁移
  1. **Phase 1**：基础设施（Vite + Phaser 初始化）
  2. **Phase 2**：渲染层迁移（Canvas 2D → Phaser WebGL）
  3. **Phase 3**：物理引擎迁移（手写碰撞 → Arcade Physics）
  4. **Phase 4**：音效系统迁移（Web Audio API → Phaser Sound Manager）
  5. **Phase 5**：视觉特效增强（粒子系统、光影效果）
- **理由**：
  - 每个阶段独立可测试，降低风险
  - 优先迁移核心功能（渲染、物理），后增强体验（特效）
  - 每个阶段完成后游戏仍可运行，便于回归测试
- **备选**：一次性全量迁移
  - 优点：开发周期短
  - 缺点：风险高，难以定位问题
  - 否决原因：影响范围大（14 个破坏性变更），一次性迁移风险不可控

## 决策 5：领域边界调整
- **选择**：保持现有领域划分不变
- **理由**：
  - 引擎迁移是技术层面的替换，不改变业务逻辑
  - CORE 领域职责清晰（游戏循环、渲染），无需拆分
  - 其他领域（PLAYER/WEAPON/ENEMY/COMBAT）与渲染引擎解耦，不受影响
- **备选**：将渲染层独立为 RENDER 领域
  - 优点：职责更单一
  - 缺点：增加领域间通信复杂度，过度设计
  - 否决原因：当前规模不需要如此细粒度的拆分

## 决策 6：性能目标设备
- **选择**：MacBook Pro M1（2020）作为最低配置
- **理由**：
  - M1 芯片性能中等偏上，覆盖大部分用户设备
  - WebGL 支持良好，可充分发挥 Phaser 性能
  - 便于开发和测试（团队主力设备）
- **性能目标**：
  - 60 FPS（100 个活跃对象）
  - CPU 占用率降低 ≥ 30%
  - 初始化时间 ≤ 3 秒
- **备选**：Intel i5 + 集成显卡
  - 优点：覆盖更广泛的低端设备
  - 缺点：性能目标过于保守，限制特效表现
  - 否决原因：赛博朋克风格需要一定的视觉效果，低端设备可通过降级方案支持

## 决策 7：音效资源格式
- **选择**：保持 Web Audio API 程序化生成
- **理由**：
  - 无需额外音频文件，减少资源体积
  - 音效参数可动态调整（音量、音调、空间位置）
  - 符合赛博朋克电子音效风格
  - Phaser Sound Manager 支持 Web Audio API
- **备选**：使用 MP3/OGG 音频文件
  - 优点：音质更好，表现力更强
  - 缺点：增加资源体积，加载时间变长
  - 否决原因：当前程序化音效已满足需求，无需引入额外资源

## 决策 8：赛博朋克角色迁移方案
- **选择**：Phaser Graphics 动态绘制
- **理由**：
  - 保持当前 2.5D 伪 3D 风格（3 面盒子绘制）
  - 无需美术资源，降低开发成本
  - 动态绘制灵活性高，便于调整细节
  - 性能开销可接受（单个角色 < 1ms）
- **备选**：预渲染 Sprite 图集
  - 优点：渲染性能更高
  - 缺点：需要美术资源，灵活性差
  - 否决原因：当前动态绘制已满足性能要求，无需引入美术资源
